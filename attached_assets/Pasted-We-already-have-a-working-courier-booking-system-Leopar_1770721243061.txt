We already have a working courier booking system (Leopards + PostEx) from Ready-to-Ship. Now implement a complete “Print & Logs” system so every booking has a printable record, and every bulk booking creates a batch log that can be reprinted/downloaded later as PDF.

GOALS (must be bug-free, audit-friendly):
1) Every single booking must generate a permanent printable “Airway Bill / Booking Slip” record.
2) Every bulk booking must generate a “Batch / Loadsheet Log” record containing all shipments booked in that action.
3) From Order Details page: user can view/print/download that order’s airway bill PDF anytime.
4) From Shipments tab: user can open batch logs, reprint/download PDFs for:
   - a single shipment
   - all shipments in a batch
5) PDFs must be consistent and downloadable. No dependence on unstable external links.

GENERAL RULES:
- Multi-tenant: everything is scoped to merchant_id.
- Idempotent: printing should not create duplicates; a shipment always points to exactly one “print record” per booking attempt.
- Durable: store print metadata + generated PDF file paths/URLs.
- Security: only authenticated users from same merchant can access PDFs.

========================================================
A) DATABASE / SCHEMA (Prisma)
========================================================
Add new tables:

1) shipment_batches
- id (uuid)
- merchant_id
- created_by_user_id
- courier_name (LEOPARDS/POSTEX)
- batch_type enum: SINGLE, BULK
- created_at
- status enum: CREATED, PARTIAL_SUCCESS, SUCCESS, FAILED
- total_selected_count
- success_count
- failed_count
- notes (optional)
- pdf_batch_path (nullable) // generated batch PDF (loadsheet)
- pdf_batch_meta (JSON) // page count, generated_at, etc.

2) shipment_print_records
- id (uuid)
- merchant_id
- shipment_id (unique)
- order_id
- courier_name
- tracking_number
- print_template_version (string) // allow template updates later
- generated_at
- generated_by_user_id
- pdf_path (string) // internal storage path
- pdf_meta (JSON) // size, pages
- source enum: COURIER_SLIP_LINK, CUSTOM_TEMPLATE
- is_latest (boolean)

3) shipment_batch_items (join table)
- id
- batch_id
- shipment_id
- order_id
- booking_status_at_time (BOOKED/FAILED)
- booking_error (nullable)
- tracking_number (nullable)
- slip_url (nullable)
- created_at

Update existing shipments table:
- batch_id (nullable)
- print_record_id (nullable)
- slip_url (nullable)
- airwaybill_number = tracking_number (or separate field if required)
- label_format enum: LEOPARDS_SLIP, POSTEX_LABEL, CUSTOM_GENERIC
- booking_payload_json (already exists if possible)
- booking_response_json (already exists if possible)

========================================================
B) PRINTING OUTPUT REQUIREMENTS (PDF)
========================================================
We need TWO types of PDFs:

1) Single Shipment PDF (Airway Bill / Slip)
- Must include:
  - Merchant/Brand name (from merchant settings)
  - Courier name + tracking number
  - Order number (#xxxx)
  - Booking date/time
  - Consignee: name, phone, city, address
  - COD amount
  - Pieces, weight
  - Items summary (optional)
  - Barcode / QR of tracking number (must be scannable)
- Must print clean on A4 and also support 4x6 if possible (toggle later).
- For Leopards:
  - If slip_link exists, we can provide “Open Original Slip” BUT we still generate our own PDF record for consistency.
- For PostEx:
  - If no official label endpoint is available, generate our own “Custom Airway Bill” PDF.

2) Batch PDF (Loadsheet / Batch Log)
- For each batch booking action, generate ONE PDF containing:
  - Batch header: Batch ID, courier name, created by, created at, totals (success/failed)
  - Table of all shipments booked in that batch:
    - Order #, Tracking #, Name, City, COD, Phone, Address (shortened), Status
  - Optional: Add page breaks after N rows
  - Barcodes for each tracking number (optional but recommended)
- This batch PDF must be downloadable and printable.

PDF generation must be done server-side in Node.
- Use a reliable PDF library (pdf-lib or puppeteer HTML->PDF). Prefer puppeteer if layout is important.
- Store PDFs in Replit storage or a DB-backed file system location, and serve via authenticated route.

========================================================
C) BOOKING FLOW CHANGES (CREATE BATCH LOGS)
========================================================
When user clicks “Book Selected” in Ready-to-Ship:

1) Create a shipment_batches record immediately with:
- total_selected_count
- batch_type = BULK if >1 else SINGLE
- courier_name
- created_by_user_id

2) During booking:
- For every order shipment created, create shipment record and link it to batch_id.
- Create shipment_batch_items rows for all selected shipments (including failed).

3) After booking completes:
- Update shipment_batches:
  - success_count, failed_count
  - status = SUCCESS / PARTIAL_SUCCESS / FAILED
- Generate batch PDF and save pdf_batch_path.

4) For each SUCCESS shipment:
- Generate shipment_print_records PDF and save pdf_path.
- Link shipments.print_record_id to latest print record.

IMPORTANT:
- If a booking is retried, do NOT create a new batch automatically unless user initiates a new booking action.
- If a shipment already has tracking_number and is booked, do not rebook; instead allow “Reprint”.
